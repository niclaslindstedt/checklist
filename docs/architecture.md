# Architecture

`checklist` is a single-page TypeScript PWA served as static files from
GitHub Pages. There is no server-side component. Every byte of user
data either lives in the browser or — if the user opts in — in the
user's own Google Drive or Dropbox account.

## Module layout

```
src/
  app/        # root component, routing, top-level state
  ui/         # presentational components and styles
  domain/     # pure functions over the data model (no I/O, no DOM)
  storage/    # pluggable persistence backends
    local/      # localStorage + IndexedDB
    drive/      # Google Drive app-folder adapter
    dropbox/    # Dropbox app-folder adapter
  share/      # encode/decode URL-fragment payloads
  pwa/        # service worker, manifest, install prompt
```

### Dependency direction

```
        ┌──────┐
        │  ui  │
        └──┬───┘
           │
   ┌───────┴────────┐
   ▼                ▼
┌──────────┐   ┌─────────┐
│  domain  │ ◄ │ storage │
└──────────┘   └─────────┘
```

- `ui/` imports from `domain/` and `storage/`.
- `storage/` imports from `domain/` for the model types only.
- `domain/` imports nothing else from `src/`, no DOM, no `fetch`.

This direction is enforced by an ESLint rule and verified in CI.

## Data model

Three types live in `src/domain/types.ts`:

- `Template` — a named list of `TemplateItem`s. Identifies itself by a
  stable `id` (UUIDv7).
- `Checklist` — an instance of a template at a point in time. Holds
  the template id, a snapshot of the items, and a per-item `checked`
  flag.
- `Item` — `{ id, title, notes?, required? }`.

Templates and checklists are stored as plain JSON. There is no
migration story yet because there are no other versions; a `version`
field is reserved on each top-level object.

## Storage

All backends implement the same interface in `src/storage/types.ts`:

```ts
interface StorageBackend {
  loadAll(): Promise<Snapshot>;
  saveTemplate(t: Template): Promise<void>;
  saveChecklist(c: Checklist): Promise<void>;
  delete(id: string): Promise<void>;
  subscribe(fn: (snapshot: Snapshot) => void): Unsubscribe;
}
```

The default backend is `LocalStorageBackend`, which writes a single
JSON document to `localStorage` under the key `checklist:v1`. The
Drive and Dropbox backends keep an analogous JSON document in their
app-folder and merge changes using a last-writer-wins strategy on the
per-item `updatedAt` timestamp.

Backends are loaded lazily. The Drive and Dropbox SDKs are not part
of the initial bundle; they are imported on demand the first time the
user selects that backend.

## Sharing

`src/share/` encodes a checklist as a gzipped, base64-URL-safe blob
and places it in the URL fragment (`#…`). Fragments are not sent to
servers, so a shared link never leaks the contents anywhere but the
recipient's browser. On import, the recipient receives a brand-new
local copy with a fresh id.

## PWA

The service worker (`src/pwa/sw.ts`) is generated by `vite-plugin-pwa`
with a `precache` strategy for the app shell and `networkFirst` for
the cloud-storage hosts. It is registered in production builds only.

## Build & deploy

`vite build` produces a fully static bundle in `dist/`. The GitHub
Pages workflow (`.github/workflows/pages.yml`) publishes it on every
push to `main`, serving the production app from the root (`/`) under
the custom domain in `public/CNAME`. The base path is injected via
`VITE_BASE` so the same bundle works at `/`, `/preview/`, `/branch/`,
or any subpath. The released `v*` tag is served at `/`, the current
`main` at `/preview/`, and an optional dispatched feature branch at
`/branch/`. See `AGENTS.md` → "Releases and changelog" for the full
release flow.
